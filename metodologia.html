Contexto e objetivo

Este documento descreve o desenho experimental para avaliar empiricamente se código gerado por LLMs (Large Language Models) tem manutenibilidade inferior à de código humano, observando a evolução de métricas de qualidade ao longo do tempo em projetos open-source com histórico misto.

Ferramentas centrais: GitHub REST/Search API (coleta de commits) e SonarQube (métricas de qualidade).

Etapa 1 — Refinar a pergunta de pesquisa

Antes de coletar dados, operacionalizar a RQ. Três possibilidades:

RQ1Código gerado por LLM tem mais code smells ao ser introduzido do que código humano?
RQ2Code smells introduzidos por LLM persistem mais tempo que os de humanos? (recomendado — alinhado com "ao longo do tempo")
RQ3Arquivos tocados por LLM acumulam dívida técnica mais rápido em commits subsequentes?

Etapa 2 — Identificar commits de IA

Assinaturas conhecidas em commits:

  • Aider: Co-authored-by: aider (gpt-4) <noreply@aider.chat>
  • Claude Code: Co-Authored-By: Claude <noreply@anthropic.com>
  • GitHub Copilot agent: autor Copilot <copilot@github.com>
  • Devin / Cursor agents: padrões próprios no autor ou co-autor
  • Copilot inline (autocomplete): não marca — invisível. Esses commits ficam no grupo "humano". Declarar como ameaça à validade.

Exemplos de busca via GitHub Search API

GET https://api.github.com/search/commits?q=author-email:noreply@aider.chat
GET https://api.github.com/search/commits?q=%22Co-authored-by:+aider%22
GET https://api.github.com/search/commits?q=author:Copilot

Header: Accept: application/vnd.github.cloak-preview+json

Etapa 3 — Seleção do projeto

Critérios objetivos (documentar na dissertação):

CritérioValor mínimo sugerido
Idade do repositório≥ 3 anos
Total de commits≥ 1000
Commits de IA identificáveis≥ 100
Commits humanos no mesmo período≥ 100
Linguagem suportada pelo SonarPython / Java / JS / TS / Go / C#
LicençaOpen-source

Estratégia prática: rodar a Search API filtrando por co-autor IA, agrupar por repositório, e selecionar os top-N que atendem aos critérios. Bons candidatos: projetos onde o mantenedor usa Aider sobre si mesmo (ex.: paul-gauthier/aider) ou onde Copilot Agent abre PRs.

Etapa 4 — Coleta de dados

Para cada commit relevante, armazenar:

  • SHA
  • Data de autoria
  • Autor (login + email)
  • Flag is_ai (boolean)
  • Arquivos modificados
  • Linhas adicionadas / removidas

Salvar em SQLite (mais simples de consultar depois que CSV). Stack sugerida: Node.js + TypeScript, com @octokit/rest (SDK oficial do GitHub, tipado, com paginação e rate-limit automáticos) e better-sqlite3. Atenção ao rate limit: 5000 req/h autenticado, 30 req/min na Search API.

Etapa 5 — Análise com SonarQube

Decisão de design crítica:

Opção A — Snapshot por commit (caro, preciso)

Para cada commit selecionado: git checkout <sha> e rodar o scanner. Um projeto Sonar por snapshot, ou sonar.projectVersion=<sha>. Permite traçar evolução fina.

Limitação: 200 commits × ~3 min ≈ 10h+ de processamento.

Opção B — Issue lifetime (recomendado)

Rodar o scanner apenas no HEAD. Cada issue tem creationDate e closeDate — mapeie smell → commit → autor (IA ou humano) e meça tempo até resolução (= persistência).

GET /api/issues/search?componentKeys=<projeto>&createdAfter=...

Combine com git blame no commit de criação para confirmar autoria. Responde a RQ2 de forma mais limpa e barata.

Etapa 6 — Métricas a comparar

Para cada grupo (IA vs humano):

  • Densidade de smells na introdução: smells por 100 linhas adicionadas
  • Tempo de vida do smell: dias entre creationDate e closeDate (censurado se ainda aberto)
  • Severidade: distribuição BLOCKER / CRITICAL / MAJOR / MINOR / INFO
  • Tipo: Bug, Vulnerability, Code Smell, Security Hotspot
  • Complexidade ciclomática dos métodos tocados
  • Duplicação introduzida

Etapa 7 — Análise estatística

  • Teste de hipótese: Mann-Whitney U (não-paramétrico) para distribuições IA vs humano
  • Análise de sobrevivência: Kaplan-Meier + log-rank para tempo até correção do smell. Análise principal da RQ2.
  • Tamanho do efeito: Cliff's delta
  • Reportar intervalos de confiança, não apenas p-valor

Ameaças à validade

Obrigatório na dissertação. Documente cada uma destas limitações:
  • Copilot inline não é detectável → grupo "humano" contaminado
  • Um commit pode misturar código IA + edição humana
  • Um único projeto = baixa validade externa → considerar 3–5 projetos
  • Regras do Sonar mudam entre versões → fixar a versão usada
  • Confounders: experiência do autor, complexidade do módulo, idade do código
